# Engineering Metrics

## Purpose

This document defines the key engineering metrics used to monitor the health, quality, reliability, and delivery performance of the Skillar Mobile project. Metrics should drive continuous improvement rather than individual performance evaluation.

---

## Guiding Principles

- Measure outcomes, not activity.
- Use metrics to identify trends.
- Balance speed with quality.
- Review metrics regularly.
- Avoid optimizing a single metric at the expense of others.

---

## Delivery Metrics

Track:

- Lead Time for Changes
- Cycle Time
- Deployment Frequency
- Release Success Rate
- Planned vs. Completed Work

---

## Quality Metrics

Monitor:

- Defect Escape Rate
- Defect Density
- Production Incidents
- Reopened Bugs
- Static Analysis Findings

---

## Reliability Metrics

Track:

- Crash-Free Sessions
- Application Availability
- Mean Time to Detect (MTTD)
- Mean Time to Recover (MTTR)

---

## Performance Metrics

Monitor:

- Cold Start Time
- Screen Load Time
- Frame Rate (FPS)
- Memory Usage
- Network Latency
- Bundle Size

---

## Testing Metrics

Track:

- Unit Test Coverage
- Integration Test Coverage
- End-to-End Test Pass Rate
- Test Execution Time
- Flaky Test Count

---

## Security Metrics

Monitor:

- Critical Vulnerabilities
- Dependency Vulnerabilities
- Security Review Findings
- Secret Scanning Results

---

## Documentation Metrics

Track:

- Documentation Coverage
- Architecture Decision Records (ADRs)
- Outdated Documentation Items

---

## Review Cadence

Engineering metrics should be reviewed:

- During sprint retrospectives
- Before major releases
- During architecture reviews
- Quarterly for long-term trends

---

## Anti-Patterns

Avoid:

- Using metrics to evaluate individuals.
- Focusing on a single KPI.
- Ignoring long-term trends.
- Optimizing metrics without improving user outcomes.
